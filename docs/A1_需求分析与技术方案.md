## 二、需求分析

### 2.1 业务背景与问题定义

目标场景：钢铁、汽车制造等行业的大型设备检修作业。

核心痛点：
- 检修知识分散在手册、日志、老员工经验中，形成"数据孤岛"
- 新人上手慢，经验传承难
- 故障响应慢，非计划停机损失大
- 作业不规范，操作失误率高

系统目标：构建一个能整合多模态检修资源、辅助一线人员快速检索方案、诊断故障、规范作业的智能助手。

### 2.2 功能需求分析

#### 功能1：多模态知识检索（核心，30%评分权重）

用户输入类型：
- 文本描述（如"发动机异响，转速不稳"）
- 故障图片（如设备损坏照片、仪表盘截图）
- 设备型号（如"LS-2000型压缩机"）

系统输出：
- 相关检修手册段落（带来源引用）
- 相似历史故障案例
- 跨模态匹配结果（图文联合检索）

关键子需求：
- 文本向量化检索（语义相似度）
- 图像特征提取与匹配
- 设备型号结构化过滤
- 检索结果排序与高亮

#### 功能2：标准化作业指引（核心）

输入：设备类型 + 检修等级 + 故障类型
输出：

- 步骤化操作指引（有序列表，含注意事项）
- 合规校验提醒（安全规范、工具要求）
- 个性化流程推送（根据设备型号和检修等级动态调整）

关键子需求：
- 作业流程模板管理
- 基于 LLM 的动态流程生成
- 合规规则嵌入（可配置）

#### 功能3：知识沉淀与更新（核心）

输入：一线人员上传的检修案例、经验总结（文本/图片/PDF）
流程：上传 → 解析 → 审核 → 纳入**知识图谱**
支持：

- 人工标注大模型输出结果（正确/错误/修正）
- 修正结果回写知识库
- 知识图谱可视化展示（设备-部件-故障-原因-步骤关系）

关键子需求：
- 文档解析（PDF、图片 OCR）
- 审核工作流（简单的人工审核界面）
- 知识图谱轻量实现
- 标注与反馈闭环

## 三、技术选型

### 3.1 选型约束

- 必须在 LoongArch + 银河麒麟 V11/V10 上运行
- Python 生态在该平台的兼容性需实测（5月中旬环境开放后验证）
- 优先选择依赖少、兼容性强的方案

### 3.2 前端

**推荐：FastAPI + Jinja2 + HTMX**

理由：
- 纯 HTML/CSS/JS，无需 Node.js 环境，麒麟兼容性最强
- HTMX 实现局部刷新，接近 SPA 体验，无需复杂前端框架
- 开发成本低，2人团队可控

备选：**Streamlit**（需在麒麟环境验证后决定）
- 优点：开发极快，适合演示
- 风险：Streamlit 在 LoongArch 上的兼容性未知，需实测

### 3.3 后端

**推荐：FastAPI + Uvicorn**

理由：
- Python 原生，LoongArch 上 Python 3.10+ 可用
- 异步支持，适合 LLM 流式输出
- 自动生成 API 文档

### 3.4 大模型接入

**推荐：云端 API（主路线）**

- Qwen（通义千问）API：官方题面明确提及，兼容性好
- DeepSeek API：官方题面明确提及，性价比高
- 优点：无需在麒麟上本地部署模型，规避 GPU 依赖问题

**备选：本地部署（加分项）**
- Ollama + Qwen2.5 / DeepSeek-R1-7B
- 需要麒麟环境支持 CUDA 或 CPU 推理
- 风险较高，建议作为加分项而非主路线

**多模态模型：**
- 图像理解：Qwen-VL API（支持图文混合输入）
- 图像 OCR：PaddleOCR（需验证 LoongArch 兼容性）或云端 OCR API

### 3.5 向量数据库

**推荐：Qdrant（客户端模式）**
- 支持 Python 客户端连接远程 Qdrant 服务
- 或使用 Qdrant 本地模式（单文件，依赖少）

**备选：ChromaDB**
- 纯 Python 实现，LoongArch 兼容性较好
- 适合快速原型

**不推荐：Milvus、Weaviate**（部署复杂，麒麟兼容性风险高）

### 3.6 关系型数据库

**推荐：PostgreSQL**
- 麒麟 V11 官方支持 PostgreSQL
- 存储设备信息、检修记录、用户标注

**备选：SQLite**（开发阶段使用，部署简单）

### 3.7 知识图谱

**推荐：轻量关系图（PostgreSQL 存储 + NetworkX 计算 + ECharts 可视化）**

节点类型：设备 → 部件 → 故障现象 → 故障原因 → 检修步骤 → 风险提示

理由：
- 不引入 Neo4j 等重型图数据库（麒麟兼容性风险）
- NetworkX 纯 Python，兼容性好
- ECharts 前端渲染，无额外依赖

### 3.8 文档解析

- PDF 解析：PyMuPDF（fitz）或 pdfplumber
- 图片 OCR：PaddleOCR 或 Tesseract
- 文本分块：LangChain TextSplitter

### 3.9 Agent 编排

**推荐：LangGraph**

- 状态机编排，适合多步骤检修流程
- 支持工具调用（检索、生成、校验）
- 比 LangChain LCEL 更适合有状态流程

### 3.10 Embedding 模型

**推荐：BGE-M3（BAAI）**
- 支持中文，效果好
- 可通过 API 调用或本地部署
- 备选：text-embedding-v3（通义千问 Embedding API）

### 3.11 技术栈汇总

| 层次 | 技术选型 | 备注 |
|------|----------|------|
| 前端 | FastAPI + Jinja2 + HTMX | 麒麟兼容性最强 |
| 后端 | FastAPI + Uvicorn | Python 异步 |
| LLM | Qwen-VL API / DeepSeek API | 云端，规避本地部署风险 |
| 向量库 | Qdrant / ChromaDB | 优先轻量方案 |
| 关系库 | PostgreSQL | 麒麟官方支持 |
| 知识图谱 | NetworkX + ECharts | 轻量，无额外服务 |
| Agent | LangGraph | 状态机编排 |
| RAG | LangChain | 检索链路 |
| 文档解析 | PyMuPDF + PaddleOCR | 需验证兼容性 |
| Embedding | BGE-M3 / 通义 API | 中文效果好 |

## 四、项目架构方案

### 4.1 整体架构

```
用户浏览器
    │
    ▼
FastAPI Web 服务（B/S 架构）
    │
    ├── 路由层（API Endpoints）
    │       ├── /search        多模态检索接口
    │       ├── /guide         作业指引生成接口
    │       ├── /knowledge     知识上传/审核接口
    │       └── /graph         知识图谱查询接口
    │
    ├── Agent 编排层（LangGraph）
    │       ├── 检索 Agent     负责多模态检索
    │       ├── 诊断 Agent     负责故障分析与作业指引生成
    │       └── 审核 Agent     负责知识入库前的质量校验
    │
    ├── 知识库层
    │       ├── 向量库（Qdrant）    文本/图像语义检索
    │       ├── 关系库（PostgreSQL）设备信息、检修记录
    │       └── 知识图谱（NetworkX）设备-故障-步骤关系
    │
    └── 模型服务层
            ├── LLM（Qwen/DeepSeek API）
            ├── VLM（Qwen-VL API）图像理解
            └── Embedding（BGE-M3）
```

### 4.3 核心业务流程

#### 多模态检索流程

```
用户输入（文本/图片/型号）
    │
    ├── 文本输入 → Embedding → 向量检索 → Top-K 结果
    ├── 图片输入 → VLM 描述提取 → Embedding → 向量检索
    └── 型号输入 → 结构化过滤 → 精确匹配
            │
            ▼
        结果融合与重排序（Rerank）
            │
            ▼
        LLM 生成综合回答（带引用来源）
            │
            ▼
        返回给用户（Markdown 渲染）
```

#### 作业指引生成流程

```
输入：设备类型 + 检修等级 + 故障描述
    │
    ▼
检索相关标准作业流程（向量库）
    │
    ▼
LangGraph 诊断 Agent
    ├── 节点1：故障分类
    ├── 节点2：检索相关案例
    ├── 节点3：生成步骤化指引
    └── 节点4：合规校验
    │
    ▼
输出：步骤化作业指引 + 注意事项 + 工具清单
```

#### 知识沉淀流程

```
一线人员上传（文本/PDF/图片）
    │
    ▼
文档解析（PyMuPDF / OCR）
    │
    ▼
LLM 结构化提取（设备、故障、步骤、结论）
    │
    ▼
审核界面（人工确认/修正）
    │
    ▼
入库（向量库 + PostgreSQL + 知识图谱更新）
```

### 4.4 知识图谱数据模型

```
节点类型：
- Device（设备）：id, name, model, category
- Component（部件）：id, name, device_id
- Symptom（故障现象）：id, description
- Cause（故障原因）：id, description
- Step（检修步骤）：id, order, content, tools_required
- Risk（风险提示）：id, level, description

边类型：
- Device --HAS_COMPONENT--> Component
- Component --SHOWS_SYMPTOM--> Symptom
- Symptom --CAUSED_BY--> Cause
- Cause --RESOLVED_BY--> Step
- Step --HAS_RISK--> Risk
```
